# Assumptions Register

Provisional product/technical decisions. Each one is revisitable; none is a verified fact until validated with the practitioner (product owner).

| ID | Assumption or unknown | Impact | Safe default adopted | Validation needed |
|----|----------------------|--------|---------------------|-------------------|
| A-01 | Single practitioner, single Windows PC is the primary deployment; macOS is now a **committed** second target (user decision 2026-07-21), Linux remains future. | Packaging, signing, secure-storage design must cover Win + macOS | Forge config builds Windows (Squirrel+ZIP) and macOS (DMG+ZIP); macOS artifacts must be built and signed on a Mac | Confirm which Mac hardware (Apple Silicon assumed) and whether an Apple Developer account exists for signing/notarization |
| A-02 | No real patient data will be entered until local authentication + at-rest encryption ship (Phase 1 completion). | Clinical safety, privacy | Code comment in `connection.ts` and this register mark production use as NOT approved yet | Practitioner sign-off on the Phase 1 security gate |
| A-03 | Spanish (es-MX) is the UI language; UI strings are currently hard-coded in Spanish. | i18n rework cost grows with every screen | Extract to i18next before Phase 2 exit | Confirm tone/formality (usted vs tú) with practitioner |
| A-04 | `sexAtBirth` is recorded (female/male/unspecified) solely for clinical calculation formulas; gender identity is a separate, consent-gated future field. | Clinical correctness of energy/anthropometry formulas | 'unspecified' allowed; formulas requiring sex will prompt rather than guess | Review wording with practitioner and privacy reviewer |
| A-05 | Duplicate patient = same first name + last name + birth date (case-insensitive) among non-archived patients. | Data quality vs false rejects | Hard CONFLICT error; archived patients excluded | May need "create anyway" override for genuine homonyms |
| A-06 | Internal file numbers are sequential integers starting at 1, assigned by the system. | Report identity, practitioner habit | Auto-assigned inside the create transaction | Practitioner may have an existing numbering scheme to migrate |
| A-07 | Timestamps are stored as ISO-8601 UTC strings; birth dates as timezone-less calendar dates. | Correctness of ages, report dates | Implemented in domain layer with tests | None — engineering decision |
| A-08 | better-sqlite3 (synchronous) is the SQLite driver; ports are deliberately synchronous. | Locked-in driver choice at the port level | Documented in ADR-0004 with migration note | Revisit only if a worker/async driver becomes necessary |
| A-09 | Encrypted-at-rest strategy is undecided: SQLCipher-compatible driver vs field-level AES-GCM envelopes (ADR-0006 pending). | Biggest open security decision | No real data until decided (A-02); key hierarchy design drafted in threat model | Spike both options on Windows AND macOS packaging before Phase 1 exit |
| A-10 | Food/nutrient datasets: no dataset is bundled yet; licensing for Mexican tables (e.g. SMAE) is unverified and must be assumed restrictive. | Legal risk if bundled without rights | Dataset manifests + adapters in Phase 4; only openly licensed data bundled by default | Legal review of every dataset license before redistribution |
| A-11 | No compliance claims (LFPDPPP, NOM clinical-record rules, HIPAA, GDPR) are made; a compliance-readiness matrix is future work. | Legal exposure if claimed falsely | README/SECURITY.md state no certification claims | Qualified legal review before any external claim |
| A-12 | Windows code-signing certificate availability is unknown. | Unsigned installers trigger SmartScreen warnings | Release docs will label unsigned builds as development builds | Practitioner/owner decision on purchasing a cert |
| A-13 | AI features are out of scope until Phase 7; no provider chosen. | None now | AI provider abstraction designed but not implemented | Provider, consent flow, and cost review at Phase 7 |
| A-14 | The practitioner performs their own backups to a user-chosen folder/drive; no cloud service is assumed. | Disaster-recovery reality | Encrypted backup container is a P0 backlog item before real data | Confirm the practitioner's actual backup habits |
| A-15 | Development happens partly under WSL2 on a Windows-mounted path; native modules built in WSL are Linux binaries. | Broken app if node_modules is shared across OSes | README instructs a fresh `pnpm install` per OS | None — environmental fact |
